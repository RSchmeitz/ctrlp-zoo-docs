Mode Inputs¶

This page is the user-facing reference for the --mode flag.

If you want the Python modules behind these behaviors, use Runtime API together with Learning API.

Important

CTRL-P still uses task for the experiment selector, for example fill, pbf_fill, or default.

The CLI execution step is now called mode.

CLI Inputs¶

These inputs come directly from ctrlp.cli:

Input

Meaning

positional experiment_path

control file under experiments/<env>/configs/<task>/<algo>/<variant>.yml

--mode

runtime mode to execute

--envs

override env count for modes that allow it

--verbose

printer verbosity

--profile

enable cProfile and write profiling.prof

--jax

force the algorithm backend to jax

--procgen

override select.procgen

Supported modes currently come from ctrlp.runner.SUPPORTED_MODES:

  • resolve

  • check

  • train

  • eval

  • demo

  • rollout

  • pretrain

  • hpo

Note

The internal compatibility module is still called ctrlp.manager.task_policy, and manager.config["tasks"] still stores the validated runtime blocks.

That is an internal naming detail. The CLI and user-facing docs should be read in terms of modes.

Runtime Blocks By Mode¶

Current validated runtime keys are:

  • backend

  • sweep

  • train

  • eval

  • rollout

  • pretrain

  • hpo

There is currently no dedicated runtime.resolve, runtime.check, or runtime.demo block. Those modes are valid, but the control-file validator will reject those keys.

resolve¶

Inputs:

  • control file path

  • select

  • runtime.sweep

  • optional CLI overrides such as --procgen and --jax

Behavior:

  • resolves paths and merged parameters

  • does not instantiate envs or models

  • does not create run folders or dataset folders

check¶

Inputs:

  • control file path

  • same resolved config surface as resolve

Behavior:

  • builds one env and one model

  • uses a fixed single-env configuration

  • validates the environment through SB3’s check_env

train¶

Primary inputs:

  • runtime.sweep

  • runtime.train

  • vec

  • reward

  • algo

  • policy

Common keys inside runtime.train:

  • timesteps

  • eval.frequency

  • eval.episodes

  • settings

  • stop

  • record

Outputs:

  • experiments/<env>/runs/<task>/<algo>/<variant>/parameters.yml

  • experiments/<env>/runs/<task>/<algo>/<variant>/agents/<agent>/best_model.zip

  • progress and evaluation CSV files under the agent folder

eval¶

Primary inputs:

  • runtime.eval

  • saved run snapshot when parameters.yml exists

Common keys inside runtime.eval:

  • episodes

  • metrics

  • sets

  • settings.load

  • variants

Behavior:

  • loads the saved model and prefers the saved snapshot when available

  • expands eval.variants into multiple evaluation variants

  • writes per-agent evaluation outputs and combined run-level CSVs

demo¶

Primary inputs:

  • control file path

  • saved run snapshot when parameters.yml exists

  • environment-side demo launcher settings when provided

Behavior:

  • no dedicated runtime.demo block today

  • uses the saved run snapshot when available

  • runs the built-in demo path or the external DAWN launcher when configured

  • falls back to headless playback automatically when no GUI display is available

rollout¶

Primary inputs:

  • runtime.rollout

  • saved run snapshot when parameters.yml exists

Common keys inside runtime.rollout:

  • mode

  • envs

  • set or sets

  • vec

Behavior:

  • rollout.mode: procgen materializes procgen datasets for built-in CTRL-P envs

  • otherwise rollout mode is inferred as heuristic for baselines and policy for trainable agents

  • forces vec.rollout_info: true so raw rollout payload is preserved for later preprocessing

pretrain¶

Primary inputs:

  • runtime.pretrain

  • dataset names from runtime.pretrain.datasets or algo.datasets

Common keys inside runtime.pretrain:

  • datasets

  • split

  • set

  • output

  • name

Behavior:

  • builds a single preprocessing env

  • resolves one or more dataset paths

  • writes the preprocessed output under data/datasets/<env>/<task>/<algo>/<output>/

hpo¶

Primary inputs:

  • runtime.hpo

  • parameters/hpo/<algo>.yml

Common keys inside runtime.hpo:

  • timesteps

  • trials

  • jobs

  • episodes

  • frequency

  • envs

  • parameters

  • objectives

  • sampler

  • pruner

  • startup_trials

  • stability

  • stability_top_fraction

  • seed

Behavior:

  • does not load the saved run snapshot

  • writes study artifacts under experiments/<env>/runs/<task>/<algo>/<variant>/hpo/

  • also writes hpo.csv, best_trial.yml, study_summary.yml, configs/<task>/<algo>/best.yml, and parameters/algo/<algo>/best.yml

Practical Rule¶

If you are unsure which inputs matter for a mode:

  1. start with Command Guide

  2. use Modes for the behavior overview

  3. use Runtime API when you want the exact Python modules behind the mode

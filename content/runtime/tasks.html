Modes¶

This page describes what each runtime mode does, when to use it, and whether it writes files.

For the short command surface, use Command Guide. For config layout, use Runtime Configuration. For all CLI inputs and per-mode config fields, use Mode Inputs. For the Python entry points behind these modes, use Runtime API.

Important

The user-facing term is mode, because the CLI entry point is ctrlp ... --mode <name>.

The experiment selector task still exists separately in paths such as experiments/<env>/configs/<task>/<algo>/<variant>.yml.

Note

Executable modes can still create missing local Gym bootstrap files on first use. resolve is the only fully side-effect-free mode.

Important

Use resolve first and check second. That catches most configuration errors before you start a real run.

Recommended Order¶

  1. resolve

  2. check

  3. train

  4. eval

Mode Details¶

resolve¶

Prints merged config, paths, and source files without building envs or models.

check¶

Builds one env and one model and validates the setup.

train¶

Runs the configured training loop and writes outputs to:

experiments/<env>/runs/<task>/<algo>/<variant>/agents/<agent>/

It also writes the frozen parameters.yml snapshot for that run so later eval and demo calls can replay the exact trained setup.

eval¶

Loads saved models, prefers the saved run snapshot when one exists, runs evaluation episodes, and writes summary outputs for each run.

Typical use:

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode eval

Tip

Use eval.settings.load: false when you want fresh evaluation monitor files. Use true only when you explicitly want to reuse existing eval outputs.

Eval variants¶

The evaluation task can expand eval.variants into a small sweep of evaluation-time changes. Each variant is normalized into its own labeled output folder under the agent eval/ directory, and the combined CSV outputs include variant columns so you can compare them later.

Typical use cases:

  • compare different vec.max_steps values

  • compare different stack depths or normalization settings

  • evaluate the same model under multiple environment-side overrides

demo¶

Runs interactive or headless demos for selected sets. Like eval, it prefers the saved run snapshot when one exists so the replay matches the trained setup.

There is currently no dedicated runtime.demo block. Demo behavior comes from the saved snapshot, environment defaults, and headless-display detection.

rollout¶

Generates policy rollout datasets under:

data/datasets/<env>/<task>/<algo>/<dataset>/

Unlike train, rollout currently sets up with load_snapshot=True, so it prefers the saved parameters.yml snapshot when one exists.

For built-in CTRL-P environments, rollout.mode: procgen can materialize ProcGen scenarios instead.

Note

The rollout mode is the normal bridge from online training to offline dataset workflows. Follow it with pretrain when you want a pretraining-ready dataset surface.

Raw rollout capture¶

When vec.rollout_info: true, CTRL-P stores raw observations and rewards before wrappers such as VecFrameStack or VecNormalize alter them. Dataset preprocessing prefers that raw rollout payload when it exists, which is important for imitation-learning and behavior-cloning datasets.

pretrain¶

Reads rollout datasets and converts them into pretraining-ready data.

hpo¶

Runs Optuna-based search and stores trial artifacts under the run root. It uses runtime.hpo together with parameters/hpo/<algo>.yml and writes generated best.yml files you can train directly afterwards.

What Has No Dedicated Runtime Block¶

These are valid modes even though they do not have matching runtime.<mode> keys:

  • resolve

  • check

  • demo

If you add runtime.demo, runtime.resolve, or runtime.check today, control-file validation will reject them.

Smoke Tests¶

Use the smoke helper when you only want structural validation:

tests/smoke/run_smoke.sh minigrid
tests/smoke/run_smoke.sh thermogrid
tests/smoke/run_smoke.sh all

Note

Use PPO for smoke validation unless you specifically need to test replay-buffer behavior.

Command Guide¶

This page explains the CTRL-P command and config structure in the order a beginner usually needs it.

If you want the full beginner workflow first, use Experiments. If you want the deeper reference pages afterwards, use Runtime Modes, Mode Inputs, and Runtime Configuration.

Start With One Command¶

Most runs use this command shape:

ctrlp experiments/<env>/configs/<task>/<algo>/<variant>.yml --mode <mode> [--envs N] [--jax] [--procgen FILE] [--verbose 2] [--profile]

For beginners, the two most important parts are:

  • the experiment path: which setup you want CTRL-P to use

  • --mode: what you want CTRL-P to do

When you run resolve, check, train, pretrain, or hpo, CTRL-P uses the editable source files under experiments/<env>/configs/ and experiments/<env>/parameters/. HPO also reads the task-local search space from experiments/<env>/parameters/hpo/<algo>.yml. When you run eval, demo, or rollout, CTRL-P prefers the saved snapshot in experiments/<env>/runs/<task>/<algo>/<variant>/parameters.yml so the replay matches the trained run.

Note

The experiment selector task and the CLI execution mode are different things. task is part of the path and select block; mode is the runtime operation passed on the command line.

Why There Are Different Modes¶

CTRL-P splits the workflow into runtime modes because inspection, validation, training, evaluation, rollout, and HPO have different needs.

CTRL-P ships runnable control files directly under experiments/<env>/configs/. For the CartPole example, the normal order is:

1. resolve¶

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode resolve

2. check¶

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode check

3. train¶

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode train --verbose 2

4. eval¶

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode eval --verbose 2

5. demo¶

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode demo

6. hpo¶

ctrlp experiments/cartpole-v1/configs/default/ppo/default.yml --mode hpo

After HPO finishes, CTRL-P writes hpo/best_trial.yml and hpo/study_summary.yml under the run folder and generates configs/<task>/<algo>/best.yml so you can train the best trial directly.

The Main Modes At A Glance¶

Mode

What it is for

resolve

inspect the final runtime setup

check

confirm env and model construction

train

train an agent and write outputs

eval

evaluate trained runs

demo

inspect behavior visually

rollout

create offline datasets

pretrain

turn datasets into pretraining inputs

hpo

run hyperparameter search

Why The Config Is Split This Way¶

The filename identifies the experiment, and the runtime block controls how each runtime mode with a dedicated block should behave. The frozen parameters.yml snapshot keeps the exact resolved settings that were used for a finished run:

runtime:
  sweep:
    seed: [0]

  train:
    timesteps: 10000

  eval:
    episodes: 100

That separation matters because training, evaluation, demo, rollout, and HPO often need different settings. It also matters because the source files are for editing, while the snapshot is for replaying an existing run.

There is currently no dedicated runtime.resolve, runtime.check, or runtime.demo block. Those modes are valid, but they are handled by runtime defaults rather than explicit YAML sections.

The Most Useful Flags At First¶

Flag

Why you would use it

positional experiment path

point CTRL-P to the experiment file to run

--mode

choose the runtime step to run

--envs

reduce or increase the number of parallel envs

--procgen

override select.procgen at the CLI

--jax

force backend selection to jax

--profile

write profiling.prof for the selected mode

--verbose 2

print more detailed progress output

Pass the experiment file path explicitly and point it at a control file under experiments/<env>/configs/.

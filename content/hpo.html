HPO¶

CTRL-P uses Optuna for hyperparameter optimization through the hpo mode.

Use Runtime Configuration for the surrounding config layout, Mode Inputs for the full --mode surface, and this page for the HPO-specific controls.

What HPO Does¶

The HPO flow:

  • reads the HPO search space from experiments/<env>/parameters/hpo/<algo>.yml

  • creates an Optuna study inside the experiment folder

  • evaluates one base trial seed per study

  • logs per-trial results and selected best parameters

Basic Command¶

ctrlp experiments/minigrid/configs/fill/ppo/default.yml --mode hpo

The HPO settings live inside runtime.hpo in the default control file for the task. HPO reuses the same configs/<task>/<algo>/default.yml entrypoint as training and reads the editable source parameters rather than an existing run snapshot. The search space itself lives in parameters/hpo/<algo>.yml.

Minimal Config Example¶

runtime:
  hpo:
    timesteps: 100000
    trials: 20
    jobs: 1
    envs: 8
    parameters: [algo]
    objectives: [reward]

Main Fields¶

Field

Meaning

timesteps

Training budget per trial

trials

Number of Optuna trials

jobs

Parallel Optuna workers

envs

Env count override for HPO trials

parameters

Parameter blocks to tune

objectives

Metrics to optimize

episodes

evaluation episodes per trial

frequency

number of evaluation checkpoints during a trial

sampler

Optuna sampler, default tpe

pruner

Optuna pruner, default median

startup_trials

number of warm-up trials before TPE / pruning becomes active

stability

include stability penalties as additional study objectives

stability_top_fraction

choose the preferred trial from the top-performing fraction before stability ranking

seed

base seed for the study; HPO does not expand runtime.sweep.seed

Where Results Are Written¶

HPO outputs live under:

experiments/<env>/runs/<task>/<algo>/<variant>/hpo/

and the parent run folder also stores:

  • hpo.csv

  • best_trial.yml

  • study_summary.yml

CTRL-P also writes a generated configs/<task>/<algo>/best.yml control file and a matching parameters/algo/<algo>/best.yml parameter overlay so you can train the best trial directly without copying values by hand.

Objective Semantics¶

Common objective names include:

  • reward

  • length

  • success

  • score

  • time

  • val_loss

Note

Multi-objective HPO is supported. In that case, the study tracks more than one optimization direction at once.

Advanced Configuration¶

Example:

runtime:
  hpo:
    timesteps: 1.0e5
    trials: 30
    jobs: 1
    envs: 8
    episodes: 10
    frequency: 10
    seed: 0
    sampler: tpe
    pruner: median
    startup_trials: 5
    parameters: [algo, vec]
    objectives: [reward, length]
    stability: true
    stability_top_fraction: 0.2
    settings:
      load: true
      save: true
      log: true
      plot: false

Samplers And Pruners¶

Current source support includes:

Key

Behavior

sampler: tpe

multivariate TPE sampler seeded from the study seed

pruner: median

Optuna median pruner with warm-up based on eval frequency

pruner: halving

successive-halving pruning

pruner: none

disable pruning

startup_trials affects both the TPE warm-up period and the median-pruner warm-up.

Stability-Aware Selection¶

CTRL-P can rank trials using both the primary objective values and a stability penalty computed from each metric history.

At a high level:

\[ ext{stability penalty} \approx \text{trend penalty} + 0.5 \cdot (\text{volatility} + \text{step noise}) \]

The preferred trial is then chosen from the top-performing fraction of completed trials, with lower stability penalties preferred inside that set.

Tip

This is useful when two trials have similar reward but one is much noisier across evaluation checkpoints.

Search Space Notes¶

  • parameters selects which config blocks are sampled, for example algo, reward, or vec

  • search ranges come from parameters/hpo/<algo>.yml

  • hpo.constraints can derive one parameter from others through simple expressions

  • hpo.constants should be rare and only pin values that are genuinely HPO-specific; if a value already lives in the default baseline, keep it there instead of repeating it here

The HPO search space lives in the dedicated task-local HPO file and is sourced separately from the frozen run snapshot. The snapshot is not the normal place to change an HPO search space.

For behavior cloning (algo: bc), HPO requires validation datasets to exist and expects a val split.

Practical Advice¶

Tip

Do not start with large HPO budgets.

Validate the base config first with:

  1. resolve

  2. check

  3. a short train

  4. only then hpo

If you want to train the best trial after HPO, use the generated configs/<task>/<algo>/best.yml control file.

Warning

If the base training run is unstable or misconfigured, HPO will not fix that. It will only search a bad space more efficiently.

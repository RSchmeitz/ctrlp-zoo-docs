Environment Guide¶

CTRL-P supports two environment categories:

Use Make Your Own Environment when you want to add a new built-in environment package and Command Guide when you only need the high-level runtime split.

  • built-in CTRL-P environments such as dawn, minigrid, and thermogrid

  • standard Gymnasium environments through a generic adapter

Shared Interface¶

All environments follow the Gymnasium API and expose the standard lifecycle:

  • __init__

  • reset

  • step

  • render

  • close

Note

The built-in environments may provide richer info fields than a generic Gymnasium environment. The runtime is designed to tolerate that difference.

Built-in CTRL-P Environments¶

Built-in environments live under:

src/ctrlp/environments/<env>/

Typical package layout:

src/ctrlp/environments/dawn/
  __init__.py
  env.py
  model.py
  defaults/

Package responsibilities¶

  • env.py: readable Gym lifecycle and env-specific behavior

  • model.py: lower-level environment model/state logic

  • defaults/: package-owned config defaults

  • __init__.py: package export and registration metadata

Generic Gymnasium Environments¶

If env: is not a registered CTRL-P environment, CTRL-P falls back to:

gymnasium.make(env_id)

Examples:

  • CartPole-v1

  • MountainCar-v0

  • other standard Gymnasium ids with supported observation/action spaces

Tip

This is useful when you want to test the runtime, training loop, or HPO flow on a standard benchmark without creating a new environment package.

Runtime Behavior¶

Reset¶

At the start of each episode:

  • the env is initialized to a valid start state

  • seed and vector-wrapper settings are applied

  • the first observation and info dict are returned

Step¶

Each step:

  1. applies the selected action

  2. advances the environment state

  3. computes reward

  4. returns observation, reward, termination flags, and info

Render¶

Depending on the environment and configuration, rendering may provide:

  • human visualization

  • RGB array output

  • environment-specific overlays or diagnostics

Info Fields¶

Built-in environments often provide fields such as:

  • score

  • procgen

  • is_success

  • physics

Generic Gymnasium environments may not.

For those environments:

  • score falls back to cumulative episodic reward

  • is_success is forwarded if the env exposes is_success or success

  • procgen is omitted when unavailable

  • physics is omitted when unavailable

Warning

Do not assume every environment exposes the same rich info dictionary. If you write new runtime code, handle optional fields explicitly.

Shared Infrastructure¶

Environment support code lives outside the concrete environment packages:

  • ctrlp.environments.common: shared environment substrate and control helpers

  • ctrlp.visualization: generic viewer infrastructure

  • ctrlp.common.environments: wrappers and vector env management

  • ctrlp.common.procgen: ProcGen support

These modules support the concrete environments but are not themselves environments.

dawn also uses task-local assets under experiments/dawn/parameters/assets/<task>/ for DAWN runtime inputs such as .pro files, control JSON, and process data. The experimental fff_warp task follows that same asset pattern.

Design¶

This page captures the main design decisions behind CTRL-P.

Use Project Layout for the package and workspace layout, Manager for resolution and execution ownership, and Environment Guide for the built-in versus Gymnasium split.

At a Glance¶

Decision

Why it exists

Experiment-first selectors

Stable inputs and stable output paths

Source and outputs are separate

Cleaner workspace layout and fewer accidental edits

resolve is read-only

Safe inspection before heavy runtime work

Built-in and Gym envs share one runtime

Easier testing and broader applicability

Run snapshots are saved

Historical runs remain reproducible and replayable

1. Experiment Selectors¶

CTRL-P organizes runs around four selectors:

  • env

  • task

  • algo

  • run

That yields a stable output structure:

experiments/<env>/runs/<task>/<algo>/<variant>/

Important

This is intentional. The folder structure is part of the experiment interface, not an incidental implementation detail.

2. Source vs Outputs¶

CTRL-P keeps these concerns distinct:

  • source code in src/ctrlp/

  • built-in defaults in src/ctrlp/environments/<env>/defaults/

  • shared environment substrate in src/ctrlp/environments/common/

  • generic viewer infrastructure in src/ctrlp/visualization/

  • task-local experiment inputs in experiments/<env>/configs/ and experiments/<env>/parameters/

  • generated outputs in env-root runs/ folders

  • offline data in data/

This prevents the editable workspace from being polluted by run artifacts.

When you want to replay an existing run, CTRL-P uses the frozen parameters.yml snapshot under runs/; when you want a new run, it uses the editable source files in configs/ and parameters/.

Area

Purpose

src/ctrlp/

framework implementation

experiments/

editable inputs

experiments/<env>/

runnable named entrypoints

experiments/<env>/runs/

generated run outputs

data/

reusable offline data

3. Read-Only Resolve¶

resolve() is read-only.

Executable modes create folders only when needed.

That matters because:

  • it reduces accidental filesystem noise

  • it keeps path behavior easier to reason about

  • it makes resolve useful for debugging configuration issues

eval(), demo(), and rollout() are replay-style modes: they prefer the saved run snapshot when one exists so their behavior matches the trained artifact.

4. One Flow for Built-in and Gymnasium Environments¶

CTRL-P supports:

  • built-in environments with package-owned defaults

  • generic Gymnasium environments through a single adapter path

This means you can:

  • work with research environments in a local workspace

  • test the runtime on standard benchmarks without creating extra source folders

Tip

Use built-in environments when you need CTRL-P-specific mechanics such as ProcGen or custom reward structure. Use Gym fallback when you want a fast runtime sanity check on a standard benchmark.

5. Native ProcGen Flow¶

For built-in CTRL-P environments, ProcGen is not treated as an afterthought.

It is integrated into:

  • environment reset

  • train/val/test scenario control

  • dataset generation

  • rollout workflows

Note

Generic Gymnasium environments do not use the built-in ProcGen path. They still support standard rollout datasets.

6. Native Sweeps¶

sweep: is a native sweep mechanism, not an external script convention.

Example:

runtime:
  sweep:
    seed: [0, 1, 2]
    learning_rate: [1e-4, 5e-4]

Each combination becomes its own run folder.

7. Reproducible Parameters¶

CTRL-P writes resolved run snapshots into:

experiments/<env>/runs/<task>/<algo>/<variant>/parameters.yml

This matters because task defaults and algorithm defaults can change over time.

Warning

If you only look at the editable config files, you may not be looking at the exact parameters a historical run used.
